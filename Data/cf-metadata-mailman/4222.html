Mike,

Here's my critique. You may not have options about some of this (I
don't know what constraints the GOES-R program has placed on you), but I
thought I would just tell you what I think.

The first thing that I noticed about this layout (schema, whatever term
you prefer) is that you don't have the actual lon-lat coordinates
represented. I understand the desire to avoid doubling up on coordinate
variables, but is this OK?

I found the "number_of_time_values" dimension name to be quite
confusing. Perhaps "number_of_time_bounds" would be better.

I also found the use of variables as containers for groups of attributes
to be troubling. I know this is done for the coordinate system
projection, but using them for generalized containers like this doesn't
conform to any standard. It looks to me like the information in the
geospatial_lat_lon variable attributes could be represented using
variables with bounds, and that the information in the
ash_cloud_height_summary_statistics and
ash_mass_loading_summary_statistics variables could be represented as
variables with associated ancillary variables. It seems to me that if
you did that, it would make the data more usable.

The product_summary_statistics variable is different, but I find myself
wondering if it wouldn't be better to have two array variables -
product_summary_statistics_names (type = string) and
product_summary_statistics_values (type = int).

Grace and peace,

Jim
